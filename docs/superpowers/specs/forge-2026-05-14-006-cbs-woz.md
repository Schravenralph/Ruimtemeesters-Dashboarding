# Forge Spec: CBS WOZ-waarde (85036NED) as a new data source

**Cycle:** TBD | **Clock:** 0 elapsed | **Size:** M-L
**Refs:** EPIC #141 (PROG-DATA: CBS Wonen data-source expansion), ADR-003
(cohort/provincie/land references on by default), ADR-006 (sync cadence
via sync_demand). Storage policy: `project_bulk_storage_to_mnt` (10 GB rule).

## What

Wire CBS table **85036NED — Gemiddelde WOZ-waarde van woningen** as a
new data source. After this spec lands, the registry has a
`cbs_woz` source and the table holds gemeente-level mean WOZ values per
year, optionally split by Eigendom (koop/huur/etc).

In scope:

- New migration `050_cbs_woz_source.sql` that:
  - **First**, runs a size-estimate calculation (in the migration
    header as a SQL comment, computed from CBS metadata) to confirm
    the table will stay <10 GB on root-disk Postgres. Rough sizing:
    ~17 years × ~380 gemeenten × ~4 Eigendom values × ~3 measure
    columns × ~50 bytes/row ≈ low MB. Comfortably under the 10 GB cap
    — root-disk Postgres is the right home.
  - Creates target table `cbs_woz` with columns `(period_year,
    region_code, region_level, eigendom_type, mean_woz_value_eur,
    source_run_id, …)`.
  - INSERTs a `data_sources` row with `key='cbs_woz'`,
    `cbs_table='85036NED'`, JSONB `sync_config` with the standard
    RegioS regionDimension (this table DOES use RegioS — confirmed in
    EPIC #141 metadata probe), `allowedLevels=['gemeente',
    'provincie', 'land']` (this is one of the few sources where
    provincie + land are useful native rows rather than computed
    rollups, given WOZ is often reported nationally), and a dimension
    mapping for `Eigendom` (T-totaal / Koop / Huur particulier / Huur
    corporatie) → `eigendom_type` column.

## Why

- WOZ is the single most-asked-for "financiële context" indicator
  for a wethouder housing dashboard — it answers "wat doet de markt in
  mijn gemeente?" in one number. Currently the Wonen supercategory has
  no price/value signal at all, which is a noticeable gap (every
  Primos-style overview surfaces WOZ or transaction prices).
- WOZ is the natural pairing with `cbs_woningen.tenure_type` (stuck
  on the CBS v3 503 outage for re-sync as of today) — Eigendom-split
  WOZ + Eigendom-split stock = a complete tenure-by-value picture per
  gemeente.
- Direct fit with Stage 1 (Wonen Primos-parity). Without WOZ the
  Wonen supercategory remains stock-and-flow only, missing the
  financial dimension.

## Success criteria

1. Migration 050 applies cleanly (idempotent re-run).
2. `data_sources` row for `cbs_woz` exists with the right
   `sync_config` (RegioS regionDimension, Eigendom mapping,
   gemeente+provincie+land levels).
3. `cbs_woz` table exists with a `(region_code, period_year,
   eigendom_type)` unique index.
4. `syncGeneric('cbs_woz', { yearFilter: 2024 })` against the v4
   endpoint returns `rowsInserted > 0` with zero errors.
5. Spot-check: mean WOZ for Amsterdam (GM0363) in 2024 in `cbs_woz`
   matches statline.cbs.nl for the same filter (Eigendom='T001038'
   total) — per `feedback_data_accuracy`.
6. Sanity range: Amsterdam mean WOZ for 2024 falls in roughly
   €400 000 – €600 000 (it should — current published values are in
   that band). Out-of-band values mean the dimension mapping is wrong
   (likely Eigendom code mismatch).
7. Cohort wiring: a query for Amsterdam with
   `references=cohort,provincie,land` returns reference values that
   are plausible (Amsterdam > G4 cohort mean > national mean is the
   typical shape).

## Approach

- Same shape as the nieuwbouw spec (#005) — config-driven, no
  per-source code path. Difference is the dimension mapping
  (Eigendom) and the measure column (mean value, not a count, so
  numeric scale + nullability differ).
- Storage check: confirm size estimate in the migration header
  comment. If reality diverges (sync row count >>> estimate), abort
  and re-plan to `/mnt/storagebox` Parquet (per
  `project_bulk_storage_to_mnt`).
- Allowed levels include provincie + land because CBS publishes these
  rows natively for WOZ (not all CBS tables do this). Keep them — the
  reference-series renderer can use them directly without computing
  weighted-average rollups.
- PR description includes the Amsterdam-2024 spot-check value vs.
  statline screenshot/link.

## Not doing

- Per-woz-klasse breakdown (CBS publishes WOZ by waardeklasse for
  some tables — not in 85036NED in the basic form, and deferred
  regardless).
- Transaction-price data (Kadaster, NVM). Different source, different
  ADR.
- Aggregating WOZ × stock for "total housing value per gemeente"
  KPI. Derivable at view-time; not stored.
- Sync schedule wiring. Demand-driven (ADR-006).
- Frontend tile/template wiring. Separate cycle.
